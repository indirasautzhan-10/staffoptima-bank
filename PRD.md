**Статус:** Draft для вайбкодинга / MVP

## 1. Обзор продукта

### 1.1 Назначение

StaffOptima Bank — аналитическая платформа для измерения достаточности или избыточности штата персонала банка и его эффективности. Продукт рассчитывает оптимальную численность под целевые бизнес-показатели, формирует KPI-систему, рекомендует набор ролей с профилями компетенций и позволяет сравнивать команды по эффективности.

### 1.2 Контекст и пользователь

Продукт создаётся для **банка** со следующей структурой:

- Три бизнес-линии: **B2C (розница)**, **B2B (корпоративный сегмент + МСБ)**, **Invest (инвестиционный бизнес)**
- Множество **продуктовых команд** (сквады/трайбы) и **цифровых подразделений**
- Целевая операционная модель — трайбы и сквады (Spotify-модель) либо гибрид

Целевые пользователи: HRD, CFO, руководители бизнес-линий, Tribe Leads, HR Business Partners, команда оргдизайна.

### 1.3 Ключевые принципы

1. **TCW-подход (Total Cost of Workforce):** эффективность считается от полной стоимости персонала, а не только от ФОТ.
2. **Бенчмарк-ориентированность:** все расчёты сравниваются с метриками международных лидеров — сингапурских банков (DBS — эталон цифрового банка) и американских игроков (JPMorgan Chase, Goldman Sachs, Stripe, Big Tech: Apple/Google/Meta), а не с классическими банками.
3. **AI-генерация рекомендаций:** анализ, KPI, структура трайбов и профили компетенций генерируются LLM (Anthropic API) на основе введённых данных.
4. **Рейтинг — инструмент диагностики, не наказания:** сравнение команд служит для выявления гипотез и точек роста.

## 2. Функциональные требования

### 2.1 Модуль «Анализ штата» (вкладка 1)

**Входные данные (сайдбар):**

| Группа | Поля |
|---|---|
| Профиль банка | Модель банка: цифровой уровня DBS (Сингапур) / универсальный (JPMorgan / Goldman класс) / классический в трансформации / финтех-Big Tech (Stripe / Apple / Google класс) |
| B2C | Доход план (млрд ₸), штат (чел.), MAU (млн) |
| B2B | Доход план (млрд ₸), штат (чел.), клиентов (тыс.) |
| Invest | Доход план (млрд ₸), штат (чел.), AUM (млрд ₸) |
| Технологии | IT-штат, число сквадов, число инженеров, релизов/мес |
| Общие | Штат всего, бэк-офис, CIR цель (%), текучесть IT (%), стратегические цели года (текст) |

**Выходные метрики (расчётные):**

- **TCW** — полная стоимость персонала, млрд ₸/год и млн ₸ на сотрудника
- **TCW / Доход** (%) — сравнение с бенчмарком модели банка
- **ROI Workforce** = (доход − TCW) / TCW
- **Доход на сотрудника** в $K (пересчёт по курсу) vs бенчмарк
- **Доля IT-штата** (%) vs бенчмарк
- **Доход/стоимость по бизнес-линиям** — TCW аллоцируется на линии пропорционально численности (с пометкой: для точности заменить на аллокацию с учётом грейдов)
- **Digital Maturity Index (0–100)** — композитный скоринг: доля IT, TCW/доход, доля бэк-офиса, релизы на сквад, текучесть IT, MAU на сотрудника. Каждое нарушение бенчмарка снижает балл.

**AI-анализ (LLM):** оценка достаточности/избыточности штата по каждой бизнес-линии с учётом TCW, окупаемость стоимости персонала, зоны избытка (кандидаты на автоматизацию) и дефицита, топ-3 решения на 2 квартала с оценкой эффекта в деньгах.

### 2.2 Модуль «Стоимость персонала» (вкладка 2)

**Входные данные — 6 статей полной стоимости (млрд ₸/год):**

1. ФОТ gross
2. Бонусы и премии
3. Налоги и социальные взносы
4. Бенефиты, ДМС
5. Лицензии ПО и рабочие места
6. Найм + обучение

**Выходы:**

- Стековая диаграмма структуры TCW с долями каждой статьи
- **Коэффициент полной стоимости** = TCW / ФОТ (ориентир ×1.35–1.55) — применяется при оценке стоимости новых вакансий и команд
- **Скрытые расходы:**
  - Стоимость текучести IT = число увольнений × ~75% годовой полной стоимости специалиста (поиск + найм + онбординг)
  - Потеря продуктивности новичков: ~6 месяцев ramp-up на 50% продуктивности
  - Простой вакансий (средний срок закрытия IT-вакансии ~3.5 мес)
  - Потери недоукомплектованных сквадов (−20–30% скорости поставки)
- Расчёт экономии от снижения текучести до бенчмарка — аргумент для бюджета удержания

### 2.3 Модуль «Сравнение команд» (вкладка 3)

**Функции:**

- Добавление команд вручную: название, бизнес-линия (B2C / B2B / Invest / Платформа), численность, полная стоимость (млн ₸/год), создаваемая ценность (млн ₸/год)
- Демо-данные (9 команд) для быстрого старта
- Удаление команд

**Расчётные метрики на команду:**

- **Efficiency Score** = Ценность / Полная стоимость
- Стоимость 1 FTE, Ценность на 1 FTE
- Рейтинг с ранжированием (медали топ-3), прогресс-бары, цветовая индикация (≥4x зелёный, ≥2.5x жёлтый, ниже — красный)
- Сводка: средняя эффективность, разброс лидер/аутсайдер

**Определение «ценности»:** прямой доход команды; для платформенных команд — аллоцированный доход или подтверждённая экономия.

**AI-анализ сравнения (по кнопке):** гипотезы о причинах разрыва (стадия продукта, недоукомплектованность, методика аллокации), корректность сравнения платформенных команд с продуктовыми и способы нормировки, действия по 2–3 наименее эффективным командам (усилить / переформатировать / слить), предостережение против использования рейтинга для наказаний.

### 2.4 Модуль «Компетенции» (вкладка 4)

**Функции:**

- Выбор роли из пресетов (14 банковских ролей): Product Owner, Backend Engineer (core banking), Mobile Engineer, Data Scientist/ML, Data Analyst, QA, DevOps/Platform, Tribe Lead, UX/UI Designer, Риск-менеджер, Комплаенс-офицер, Клиентский менеджер B2B, Менеджер по продажам Invest, Scrum Master
- Ввод произвольной роли (включая контекст трайба, например «Backend Engineer в сквад Антифрод»)

**AI генерирует профиль компетенций:**

- 8–10 компетенций на роль
- **Вес важности в % (сумма = 100%)** — визуализация горизонтальными барами
- Тип каждой компетенции: **Hard / Soft / Domain** (банковская специфика: регуляторика, риски, платёжные системы)
- Описание, индикатор сильного уровня («что отличает сильного от среднего»), метод оценки
- Сводка распределения весов по типам

**Применение весов:** скоринг кандидатов при найме (Σ оценка × вес), performance review, программы развития.

### 2.5 Модуль «KPI-конструктор» (вкладка 5)

AI генерирует **12 KPI** эффективности штата с учётом полной стоимости персонала. Обязательный набор:

| Категория | KPI (примеры обязательных) |
|---|---|
| Стоимость персонала | TCW/Доход, TCW на FTE по линиям, стоимость текучести, cost per hire |
| Эффективность штата | ROI Workforce, доход на FTE по бизнес-линиям, ценность на FTE по командам |
| Продуктовая скорость | Time-to-market, релизы на сквад/мес, time to fill |
| Клиентские | MAU на сотрудника, NPS |
| Таланты | Текучесть ключевых ролей, eNPS, span of control |

Для каждого KPI: название, категория, формула, **таргет на основе бенчмарков**, частота замера, владелец, описание.

### 2.6 Модуль «Бенчмарки» (вкладка 6)

Справочная таблица бенчмарков по выбранной модели банка с рыночным контекстом:

| Метрика | Цифровой банк (DBS-класс) | Контекст |
|---|---|---|
| TCW / Доход | 15–25% | У эффективных международных игроков < 25% |
| Доля IT в штате | 30–40% | DBS ~1/3 штата в tech (инсорсинг 90%+), JPMorgan ~57K технологов (~20%), Big Tech 50%+ |
| Доход на сотрудника | $350–550K | Goldman Sachs ~$1.1M, JPMorgan ~$520K, DBS ~$400K; Big Tech: Apple ~$2.4M, Google ~$1.9M |
| CIR | 38–42% | DBS ~39–40% — лучший в классе для универсального банка; JPMorgan ~55%, Goldman ~62% |
| Доля бэк-офиса | 20–30% | DBS автоматизировал операции через AI/ML; цель < 30% |
| Размер сквада | 7–9 чел. | Amazon two-pizza team; Spotify squad 6–12; DBS platform operating model |
| Инженеров на сквад | 4–6 | Big Tech: инженеры:продакты ~6–8:1; банки ~5:1 |
| MAU на сотрудника | 300–800 | Stripe и Big Tech 2000+, цифровые банки 300–800, классика < 400 |
| Релизов/сквад/мес | 2–8 | Elite DORA (Google/Amazon) — deploy on demand; DBS — недельные циклы |
| Текучесть IT | 10–15% | Big Tech и сингапурские банки удерживают 10–15%; > 20% — потеря экспертизы |
| Коэффициент полной стоимости | ×1.35–1.55 от ФОТ | В США с бенефитами и equity до ×1.6–1.8 (Big Tech) |
| Стоимость замены IT-специалиста | 50–100% годовой стоимости | Поиск + онбординг + ramp-up |
| Span of control | 6–8 прямых | Google/Meta для EM до 8–10; Amazon — two-pizza ownership |
| QA : инженеры | 1:3–4 | Big Tech при зрелых автотестах 1:6–10 (SDET-модель) |

Бенчмарки заданы для 4 моделей банка: digital (DBS-класс), universal (JPMorgan/Goldman класс), classic (банк в трансформации), neo (Stripe/Big Tech класс) — и должны храниться в конфигурации, доступной для редактирования. **Дисклеймер обязателен:** ориентиры на основе публичной отчётности, перед презентацией правлению сверять со свежими годовыми отчётами.

**Эталонный состав сквада** (для расчётов в модуле команд):

- 1 × Product Owner
- 2.5 × Backend Engineer
- 1.5 × Frontend/Mobile Engineer
- 1 × QA Engineer
- 0.4 × Designer (шаринг 1:2–3 сквада)
- 0.5 × Data Analyst (шаринг)
- 0.3 × DevOps (платформенная команда, 1:3–4 сквада)

Итого ~7.2 FTE на сквад. Расчёт оптимального числа сквадов = инженеры / середина бенчмарка инженеров-на-сквад.

## 3. Технические требования

### 3.1 Стек

- **Frontend:** одностраничный HTML + vanilla JS + CSS (тёмная тема), без сборки — открывается в браузере
- **Шрифты:** Inter (UI), JetBrains Mono (цифры/метрики)
- **AI:** Anthropic API `/v1/messages`, модель `claude-sonnet-4-20250514`, max_tokens 1000
- **Хранение:** состояние в памяти сессии (без localStorage); опционально — экспорт/импорт JSON

### 3.2 AI-интеграция

- Все AI-вызовы получают единый **контекст банка** (bankContext): модель, штат, TCW с разбивкой, бизнес-линии, цели, бенчмарки
- Структурированные ответы (KPI, трайбы, компетенции) запрашиваются **строго в JSON** без markdown; ответ очищается от ```json-фенсов и парсится с try/catch
- Системные промпты задают роль эксперта (оргдизайн цифровых банков / KPI / компетенционные модели) и язык ответа (русский)
- Сумма весов компетенций должна равняться 100% (валидируется в промпте)

### 3.3 UX-требования

- 6 вкладок в шапке, сайдбар ввода 340px, основная зона с карточками
- Пустое состояние до первого анализа с пояснением
- Параллельная загрузка AI-блоков (Promise.all), спиннеры с pulse-анимацией
- Цветовая семантика: зелёный = в бенчмарке, жёлтый = пограничное, красный = вне нормы
- Обработка ошибок API с понятным сообщением в карточке

## 4. Расчётные формулы (спецификация)

```
TCW = ФОТ + Бонусы + Налоги + Бенефиты + Лицензии/места + Найм/обучение
Коэффициент полной стоимости = TCW / ФОТ
TCW на FTE = TCW / Штат всего
TCW линии = (TCW / Штат всего) × Штат линии        // v1: пропорционально, v2: по грейдам
ROI Workforce = (Доход − TCW) / TCW
Доход на сотрудника ($K) = Доход(₸) / Курс / Штат / 1000
Efficiency Score команды = Ценность / Полная стоимость команды
Стоимость текучести IT = (IT-штат × Текучесть%) × 0.75 × Полная стоимость IT-специалиста
  где Полная стоимость IT-специалиста ≈ TCW на FTE × 1.6
Потери ramp-up = Увольнения × Полная стоимость × 0.5 (доля года) × 0.5 (потеря продуктивности)
Оптимум сквадов = Инженеры / mid(бенчмарк инженеров на сквад)
Digital Maturity Index = 100 − Σ штрафов за нарушение бенчмарков (18/18/14/14/12/12), floor 20
```

## 5. Ограничения и риски

1. **Аллокация TCW по линиям пропорционально численности** искажает картину при разной структуре грейдов — в v2 нужна аллокация по грейдам/ABC.
2. **Ценность платформенных команд** не измеряется прямым доходом — требуется методика аллокации (verified-экономия, internal pricing), иначе сравнение некорректно.
3. **Риск Гудхарта:** рейтинг команд не должен использоваться для наказаний — иначе команды оптимизируют метрику, а не ценность. AI-анализ обязан содержать предупреждение.
4. **Устаревание бенчмарков:** цифры по публичным компаниям меняются; перед использованием в управленческих решениях сверять с актуальной отчётностью.
5. **AI-выводы — гипотезы, не вердикты:** все рекомендации требуют валидации с руководителями линий и HRBP.
